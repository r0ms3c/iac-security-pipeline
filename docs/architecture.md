# Architecture

## Overview

The IaC Security Pipeline follows the same shared-pipeline pattern as the application security pipeline. One central repository contains the pipeline logic. All infrastructure repositories contain only a 3-line stub Jenkinsfile pointing at it.

```
GitLab group: security-pipeline/
├── security-pipeline/         ← application security pipeline
└── infra-security-pipeline/   ← this repository
```

## Components

| Component | Role |
|---|---|
| **Jenkins** | Executes the pipeline on every push |
| **GitLab** | Source code hosting + webhook trigger + commit comments |
| **Trivy** | CVE scanning + misconfiguration + secrets detection |
| **Checkov** | IaC policy compliance across all supported formats |
| **ansible-lint** | Ansible playbook quality and security |
| **tflint** | Terraform syntax and best practice validation |
| **Hadolint** | Dockerfile best practice linting |

## Data flow

```
1. Developer pushes to infrastructure repository
2. GitLab fires webhook → Jenkins job
3. Jenkins checks out infra-security-pipeline (this repo)
4. Copies helper scripts to /tmp
5. Checks out the target infrastructure repository
6. Detects IaC types from file contents
7. Runs applicable tools
8. Generates 3 HTML reports
9. Posts findings as GitLab commit comments
10. Updates commit status (pass/fail)
11. Archives HTML reports as Jenkins build artifacts
```

## Tool execution order

Trivy and Checkov always run. The three linters run only if the corresponding IaC type is detected:

```
Trivy    → always (covers CVEs + misconfigs + secrets for all types)
Checkov  → always (covers Terraform, Ansible, K8s, Compose, Dockerfile)
ansible-lint → only if HAS_ANSIBLE=true
tflint   → only if HAS_TERRAFORM=true
Hadolint → only if HAS_DOCKERFILE=true
```

## Shared pipeline pattern

The central Jenkinsfile lives in `infra-security-pipeline/Jenkinsfile`. Infrastructure repositories contain only:

```groovy
// Infrastructure security pipeline — managed centrally by the security team.
// Do not modify this file. Contact the security team for pipeline changes.
// Pipeline source: <gitlab>/security-pipeline/infra-security-pipeline
```

When the Jenkins job runs, it:
1. Reads the Jenkinsfile from **this repository** (not the infrastructure repo)
2. Copies the helper scripts to `/tmp`
3. Replaces the workspace with the infrastructure repository
4. Executes the pipeline against the infrastructure code

Any change to the central Jenkinsfile applies to all infrastructure repositories on their next push. No per-repository changes needed.

## Report generation

Three HTML reports are generated by `generate-infra-report.py`:

| File | Audience | Content |
|---|---|---|
| `infra-security-report.html` | Security team · Management | Executive summary, counts per tool, scan metadata |
| `trivy-detail.html` | Security team | Full CVE list, all misconfigurations, secrets |
| `checkov-detail.html` | Security team | All policy violations by framework and file |

Reports are archived as Jenkins build artifacts and accessible from Jenkins → job → build → Build Artifacts.
